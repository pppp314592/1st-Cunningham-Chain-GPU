# 進捗ログ

## 2025-06-25: GPU カニンガム鎖探索 修正・テスト

### 問題点
- `prime_filter.jl` の `mulmod` で `a <<= 1` が Int64 オーバーフローを起こす
  - `m > 2^62` のとき `a + a` が符号ビットに到達 → 負数にラップ → Miller-Rabin が誤判定
- `cc_gpu.jl` のオーバーフロー防止チェックが過剰
  - `x > 2^62` で即 return false → 本来は Int64 範囲内の有効なリンクまで確認可能
- CC13/14 の既知鎖は途中リンクが Int64 上限を超えるため GPU のみでは検証不能
  - CC13(4.09e15): 13番目のリンクが Int64 超過
  - CC14(9.54e16): 7番目以降 Int64 超過

### 修正内容
1. ✅ `mulmod` を UInt64 内部演算に変更 (`a <<= 1` → `au += au`) — オーバーフロー解決
2. ✅ `cc_check_kernel` を `cc_count_kernel` に変更: 確認できたリンク数を返す
3. ✅ `filter_cc_gpu` に CPU フォールバックを追加: Int128 + `Primes.isprime` で残りリンク検証
4. ✅ 篩の `n > p` 条件追加 (素数 p が候補 n 自身を篩い落とす問題を修正)
5. ✅ テストの CPU 比較関数が偶数スタートしていたバグを修正

### 検証結果 (確定)
- CC12(554688278429): ✅ PASS
- CC13(4090932431513069): ✅ PASS (CPU フォールバック)
- CC14(95405042230542329): ✅ PASS (CPU フォールバック)
- CC5 [1, 100000): GPU 7件 = CPU 7件 ✅
- CC5 [1, 500000): 篩あり20件 = 篩なし20件 ✅
- CC5 [1, 1_000_000): GPU 32件 = CPU 32件, 加速比 3.9x

### 実装内容
**128-bit ソフトウェアエミュレーション (`prime_filter.jl` に追加)**
- `add128` / `sub128` / `ge128` / `is_zero128` — 基本演算
- `mod_add128` — 桁上がり検出付き 128-bit 加算＋モジュロ
- `mod_double128` — 2倍＋モジュロ (同)
- `mulmod128` — ロシア農民法 128-bit
- `powermod128` — バイナリ指数乗
- `is_prime_mr128` — 7基底 Miller-Rabin for < 2^128

**設計のポイント**
- 桁上がり (`overflow`) 検出時に `adj = 2^128 - m` を加算する公式を導出
  - `wrapped + (2^128 - m) = (r + a) - m < m` となり再減算不要
  - `m > 2^127` のとき成立 (オーバーフロー発生条件と一致)
- `cc_count_kernel` は Int64 範囲のリンクには従来の高速 MR、超えたら `is_prime_mr128`
