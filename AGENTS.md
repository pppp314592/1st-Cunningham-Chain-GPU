### 2025-06-25: GPU 並列素数フィルター (`prime_filter.jl`)

**概要:** 整数配列を受け取り、GPU (CUDA.jl) で並列に素数判定して素数だけ返す関数。

**ファイル:**
- `prime_filter.jl` — 本体（`filter_primes(arr)` がエントリポイント）
- `test_primes.jl` — 小型〜中規模テスト
- `test_large.jl` — 大規模テスト (`10^7:10^8`)

**アプローチ:** 試し割り → Miller-Rabin 確率的素数判定（決定版）

**実装の変遷:**
1. **試し割り (6k±1)** — 遅い。10^9 オーダーでは Primes.jl の篩に勝てない
2. **`powermod` を使った Miller-Rabin (Int64)** — `Int128` 非対応で `__divti3` エラー
3. **`mulmod` 自作 + `powermod_gpu` 自作 (Int64)** — 誤った結果（`%` が GPU で不正だった可能性）
4. **`Int32` 版** — コンパイルエラー（タプル反復が GPU で MethodError）
5. **最終版 (Int64, 爆弾回避済み):** ✅ 正解一致

**最終版のポイント:**
- `rem_gpu(a, b) = a - (a ÷ b) * b` — `%` 演算子を避ける
- 7 基底 [2, 325, 9375, 28178, 450775, 9780504, 1795265022] を **ループ展開**（タプル不使用）
- `mulmod` はロシア農民法（倍々加算）でオーバーフロー回避
- `Int64` で動作

**テスト結果:**
- `10^7:10^8` (9000万要素): GPU 5,096,876 件, Primes.jl と一致 ✅

**検証コマンド:**
```julia
include("prime_filter.jl"); using Primes
filter_primes(collect(10^7:10^8)) |> length
primes(10^7, 10^8) |> length
```

### 2025-06-25: GPU カニンガム鎖探索 (`cc_gpu.jl`)

**概要:** notebook の篩アルゴリズムと GPU (CUDA.jl) Miller-Rabin を統合し、第1種カニンガム鎖を高速探索。

**ファイル:**
- `cc_gpu.jl` — 本体（`search_cc_gpu(lo, hi, target_cc)` がエントリポイント）
- `test_cc_gpu.jl` — テスト

**アーキテクチャ:**
- **Phase 1 (CPU):** Wheel 篩 [2,3,5,7,11,13] + modulo 篩で候補を削減（notebook の MySieve 相当）
- **Phase 2 (GPU):** バッチ単位で `CuArray.map` により全候補のチェインを並列判定

**GPU カーネル:** `cc_count_kernel(n, target_cc)`
- 各スレッドが 1 つの数 n について Miller-Rabin を `target_cc` 回直列実行
- `Int64` 範囲のリンクには高速 `is_prime_mr`、超過後は `is_prime_mr128` (128-bit ソフトエミュ)
- オーバーフロー検出: `x_hi > 2^63` で打切り (128-bit 上限)

**128-bit ソフトエミュレーション (`prime_filter.jl` 内):**
- 2個の `UInt64` (lo, hi) で 128-bit を表現
- 桁上がり検出時、`2^128 mod m = 2^128 - m` を加算する公式により再減算不要
- 7 基底 Miller-Rabin をフル 128-bit 対応 (`is_prime_mr128`)

**篩の効果:**
- 篩あり: 候補数が約 1/5000 に削減 (CC5 の場合)
- 篩なし: 奇数全列挙 (GPU に多く委譲)

**検証結果:**
- CC12(554688278429): ✅ GPU のみ
- CC13(4090932431513069): ✅ GPU のみ (Int64 超過リンクを 128-bit MR で処理)
- CC14(95405042230542329): ✅ GPU のみ
- CC5 [1, 1_000_000]: GPU 32件 = CPU 32件, 3.7x 高速化

**検証コマンド:**
```julia
include("cc_gpu.jl")
search_cc_gpu(1, 10^6, 5)
```

### 2025-06-25: ベンチマーク・マルチスレッド化

**概要:** notebook (CPU, i5-3330, 4T) と GPU 版の CC13 探索時間を比較。篩ループをマルチスレッド化。

**notebook の元の計測値 (CPU, 4T):**
- 10^13〜10^14: 14.59秒
- 10^14〜10^15: 148.91秒
- 4e15〜5e15: 163.19秒 → CC13: 4090932431513069 発見

**GPU 版の課題:**
- 篩 (Phase 1) がシングルスレッドのため、notebook の `Threads.@threads` 版より低速
- 候補数が少なく (wheel 効率 ~19万倍)、GPU の並列性が活きていない
- 初回コンパイル + JIT のオーバーヘッドも大きい

**修正:**
- `search_cc_gpu` の篩ループを `Threads.@threads` で並列化
- スレッドごとにローカル結果配列を持ち、最後に `vcat` でマージ
- 進捗表示は `Threads.Atomic{Int}` でカウント

**ベンチマークファイル:**
- `bench_cc13.jl` — CC13 探索の GPU vs CPU 比較

**検証コマンド:**
```julia
julia -t 12 bench_cc13.jl
```

### 2025-06-25: GPU 篩の本格実装 + Int128 対応

**背景:**
- CPU 篩がボトルネックで GPU 使用率が 2% 未満だった
- Int64 上限 (~9.22e18) で探索が打ち止め

**変更内容:**
1. **篩の GPU カーネル化** (`_sieve_kernel64`)
   - 各サイクルの 28M 要素のコピー＋40回のフィルターを 1 回の GPU カーネル起動に統合
   - 原子カウンタ (`CUDA.atomic_add!`) で動的出力
   - **9.7倍高速化** (CC15: 155s→16s/batch)
2. **篩のキャッシュ** (`_get_cached_sieve`)
   - wheel_n, primes, badset を一度 GPU に転送して使い回し
3. **Int128 対応** (`search_cc_gpu128`, `_sieve_kernel128`)
   - ベースアドレス `k*wheel` を 128-bit で計算
   - GPU 上で `rem128by32` により 128-bit 剰余をソフトウェア実装
   - `filter_cc_gpu128` で Int128 候補の MR 判定
4. **`n <= p` ガード** — n が篩素数以下のときは自動通過（素数自身を除外しない）

**テスト結果:**
- CC5 [1, 100000): 7件 = CPU 7件 ✅
- CC5 [1, 500000): 20件 ✅
- CC5 [1, 1_000_000): 32件 = CPU 32件, 3.2x ✅
- CC12/13/14: ✅
