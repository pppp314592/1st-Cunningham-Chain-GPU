# A9 — 分散処理・チェックポイント戦略

巨大数 CC 探索は「候補が多い・MR が重い」ので、単機では時間がかかる。
A5 のプリモリアル・バンド法は、そのまま分散可能になる。

## 仕事の分割単位

1. **残基（wheel 生存 r）ごとに分割**: `M = ∏ wheel` の各 r は独立した
   等差数列 `m = r + t·M`。r の数は `φ_wheel` 通り。各ワーカに r の集合を
   割り当てる。最も粗い分割単位。
2. **t ブロックごとに分割**: 各 r ごとに `t` を帯（バンド）に切り、ワーカへ。
   バンド幅は「MR 候補数 × MR コスト」が許容になる大きさに。
3. **MR 候補ごとに分割**: 篩生存者だけを集めてキューにし、ワーカが MR を
   消費する。篩（CPU/GPU 一括）と MR（多数ワーカ）を分離する形。

## チェックポイント

* バンド探索は「r ごと・t ブロックごと」に独立なので、完了した (r, t_block)
  をログへ書き、落ちたらそこから再開できる。
* 部分結果（見つかった鎖）は即座に append し、重複は後で sort/unique。
* 確定チェーンは BPSW（A8）で最終確認し、さらに独立実装（別ワーカ）で
  クロスチェックすると偽陽性を排除できる。

## なぜこれが効くか

* 篩は「小素数で割り切れない」という安い判定。A3 のインクリメンタル更新で
  各ブロックが独立に篩えるので、ワーカ間で通信不要。
* MR だけが重いが、生存者は篩で激減している（A6 の W_k が効いている）。
  だから MR を並列化するのが最も効く。
* 既存 `cc_gpu_wheel.jl` は「1 スレッド = 1 ホイール候補」で GPU 並列していた。
  これを A5 の r 分割に置き換え、バンドを GPU の仕事単位にすれば、
  巨大数でも同じ並列性が得られる。
