# A10 — 位数 ord_p(2) による「被覆」と篩の優先順位

## 核心の観察

m 形式の悪残基は `BadSet_q = { (2^i)^{-1} mod q : i = 0..k-1 }`。
大きさは `|BadSet_q| = min(k, ord_q(2))`（`ord_q(2)` = 2 の mod q での位数）。
つまり**各素数 q は、高々 `ord_q(2)` 個の異なる i しか「殺せない」**。
しかもそれらは `i mod ord_q(2)` のサイクルで現れる。

## 篩で「強い」素数

* `ord_q(2)` が小さい素数（q = 3,5,7,11,13,17,31,73,…: 2 の位数が小）は、
  少ない残基で多くの i をカバーし、かつサイクルが短いので候補をよく絞る。
  → wheel に入れる価値が高い。実際 `src/cc_gpu.jl` の `build_cc_sieve` も
  `wl = [2,3,5,7,11,13,17,19,23,37,41,…]` と小さい素数を優先している。
* `ord_q(2)` が q−1 に近い（原始根）素数は、k が小さいうちはほぼ k 個の
  異なる残基を生成し、篩の「密度削減」への寄与は薄い（1 ブロックに 1 個の
  悪残基が増えるだけ）。ただし上限 P を大きくするほど確定 MR が減るので、
  大きい q も「上限までの積」としては効く。

## 重み W_k の急速な増大との関係（A6 の補足）

W_k が k と共に急増するのは、p=3,5,7 のような小位数素数の因子
`(1−1/p)^{−k}(1−ν/p)` が大きいため（A6 の表参照: W_2≈1.3 → W_15≈3.2×10^6）。
裏を返すと、**これら小位数素数でしっかり篩うほど、残る候補が「本当に疎」に
なり、MR コストが下がる**。だから wheel（A4）には小位数素数をなるべく多く、
追加篩（A5 の extra）でも小位数素数を優先的に厚くするのが最適。

## 応用: 「どの素数まで篩うか」の目安

* 篩上限 P を 2 倍にすると、候補密度はおよそ `∏_{p∈(P,2P]}(1−ν(p)/p)` 倍に。
  小位数素数が含まれる帯ではこの削減が大きい。
* 実測（STATUS の通り）高 CC では篩が支配的。よって P は GPU/CPU の
  メモリと転送に収まる範囲で「できるだけ大きく」が正解。A7 の 32bit 制約
  （P ≤ 2^31）は、その上限を決める現実的ライン。
