# A14 — バケツ篩（bucket/word-packed sieve）でキャッシュを効かせる

A3 の segmented sieve は「各素数 p ごとに全ブロックをなめる」ので、p が多くなると
メモリを辿る回数が `P / log P × ブロック数` になる。P が大きい（巨大数では必然）と
L2/L3 キャッシュを逸脱し、DRAM 待ちが支配的になる。

## バケツ法（bucket sieve）

各素数 p を「小・中・大」に分け、篩の単位を変える：

1. **小素数（p ≤ √B 程度）**: 従来通りブロック内で `p` 刻みに直接抹消。
   ブロック長 B より p が小さいので 1 ブロック内に複数回現れ、キャッシュに載る。
2. **中・大素数（p > √B）**: 1 ブロック内に高々 1 回（または 0 回）しか現れない。
   このとき「各ブロックごとに、落とすべきインデクスをバケツ（リスト）に蓄え、
   後で一気に抹消」する。素数をブロック全体で共有せず、出現ブロックだけ処理する
   ので無駄が減る。
3. **巨大素数（p が B より大きい）**: 出現するブロックは 0 または 1。該当ブロック
   だけに悪残基を書き込む。

## ワード圧縮

sieve を `BitVector` ではなく `UInt64` のビット列（1 語 = 64 候補）にし、
`|=` マスクで一気に複数抹消。さらに「生存ビット」をポピュレーションカウントで
数え、生存者走査を枝刈りする。GPU 版（`cc_gpu_wheel.jl`）は 1 スレッド=1 候補
だが、CPU 側ではワード単位の方がメモリ帯域で有利。

## メモリ階層の使い分け（A19 と併せて）

* L1/L2 に載る小ブロックで小素数を回し、RAM 上のバケツで大素数を捌く。
* これで「P を大きくする（= MR を減らす）」ほど篩が重くなる傾向を和らげ、
  A7 の「P ≤ 2^31 で厚く篩う」方針を実機で支える。
