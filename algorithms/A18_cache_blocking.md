# A18 — キャッシュブロッキングとブロックサイズの選択

A3 のブロック長 `B` は単に「メモリに収める」以上の意味を持つ。最適値は
ハードウェアで決まる。

## 指針

* **B を L2 キャッシュに収める**（例: 数 MB）と、小素数の `p` 刻み走査が
  L2 ヒットし、DRAM 待ちが減る。ただし B が小さすぎるとブロック数が増え、
  インクリメンタル更新（`bmod`）の回数が増える。トレードオフ。
* **B を 2 の冪（1<<22 等）にする**と、`mod(B, p)` や `idx` 計算が安い。
* 実測で `B` を 2^20 / 2^22 / 2^24 で比較し、スループットが最大になる値を採用。
  STATUS の `bench_*` 系と同様のマイクロベンチを `algorithms` 用にも用意すると良い。

## 並列とブロックの割り当て（A9 と併せて）

* ブロックをスレッドに「連続チャンク」で割る（A3 の `process_chunk!` 方式）。
  各スレッドが `bmod` を局所保持し、ブロック間でインクリメンタル更新するので
  共有状態なし。これが A3 の並列の鍵。
* NUMA 環境では、各スレッドが扱うブロックの基数 `nlo + (b-1)*B` をスレッドの
  ローカルメモリに置くと更に速い（巨大数では base は BigInt 1 個なので軽い）。

## プロファイルすべき場所

1. 小素数篩（ブロック内ループ）の CPU サイクル率。
2. 生存者収集（sieve 走査）のコスト。
3. MR（BPSW, A8）のコスト — 高 CC では篩が支配的（STATUS の分析通り）。
高 CC では (1) が支配的になるので、A14 のバケツ法で (1) を削るのが正解。
