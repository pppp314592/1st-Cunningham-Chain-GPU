# 第一種カニンガム鎖 GPU 探索 (1st Cunningham Chain GPU)

Julia + CUDA による**第一種カニンガム鎖**（Cunningham chain of the first kind）の GPU 探索。

第一種カニンガム鎖とは、ある整数 n に対し

```
n, 2n+1, 4n+3, …, 2^(k-1)·n + (2^(k-1)-1)
```

がすべて素数になる列のこと。本プロジェクトでは鎖の「先頭」 n を GPU で全探索し、
最小の n を求める。

---

## 成果（最小の先頭 n）

| k (鎖長) | 最小の先頭 n | 備考 |
|---|---|---|
| 12 | `554688278429` | (Primes.isprime で全要素確認) |
| 13 | `4090932431513069` | 同上 |
| 14 | `95405042230542329` | 同上 |
| 15 | `90616211958465842219` | **[2, 1e20] を全探索して確定**（2026-07-13） |
| 16 | `810433818265726529159` | OEIS A005602 と一致 |

CC15 は `scan.jl` で n=2 から昇順に全探索（`[2, 9.06e19]`: 10.91h、`[9.06e19, 1e20]`: 1.5h）し、
それ未満に鎖が存在しないことを確認した。

---

## 実装の要点

- **Int128 頭カーネル**: n を Int128 で生成・走査し、Int64 の限界 (≲9.2e18) を超える領域（CC15/16）にも到達。
- **拡張ホイール (ext, 既定)**: 素数 2,3,5,7,11,13,17,19,23 に **29 を追加**し、候補密度を約 ×0.448 に削減
  （materialize 方式、n < 9.06e23 まで）。
- **ストリーミング CRT (stream)**: 31, 47 をさらに追加した巨大ホイール (6.15e17) を、候補を列挙せず GPU 内で
  **その場合成**（odometer + CRT）。大レンジ (width ≫ 6e17) 走査で ~1.3–2× 高速。
- **Barrett 還元の篩**: 64bit `% p` を `mul+shift` に置換（GPU は HW 整数除算が遅いため）。篩フェーズを **2.34×**。
- **占有率チューニング**: `threads/block = 64` が最適（高レジスタカーネルで SM 占有率が敏感）。**1.26×**。
- **hot/cold パス分割 (篩/MR 分離)**: 篩専用カーネル (24reg) で生存候補を蓄積し、稀に MR をドレイン。
  融合カーネル比 **1.37–1.42×**。
  - ⚠ タイル毎のドレインは非同期パイプラインにバブルが出て利得消滅 (1.0×)。蓄積→稀ドレインが鍵。

累積高速化（旧 fused@256 基準）: Barrett 2.34× × 占有率 1.26× × 分割 ~1.4×。

### 実行環境
- Julia 1.12.3 / **CUDA.jl 5.10.0**（CUDA 13.1 環境）
- NVIDIA GPU（開発: GTX 1660 Ti / sm_75 Turing）。スループット ~1.9×10¹⁵ cand/s（CC15, ext）
- **起動時は `-g0` が必須**: フォルダパスが非 ASCII（日本語）のため、line-info に埋め込まれた文字で
  ptxas が "Unexpected non-ASCII character" で失敗する。`-g0` で回避。

## 使い方

```powershell
# 本番走査スクリプト（チャンク分割 + チェックポイント/再開 + CPU 独立検証）
julia -g0 -t 12 scan.jl <k> <start> <width> [chunk] [ext|stream] [--fresh]

# 例: CC15 を [2, 1e18) だけ試走
julia -g0 -t 12 scan.jl 15 2 1000000000000000000

# 例: CC16 を既知最小付近の帯域で確認
julia -g0 -t 12 scan.jl 16 810000000000000000000 10000000000000000000
```

- 中断しても `--fresh` なしで再実行すれば `logs/scan_k<k>.ckpt` から自動再開。
- 発見した鎖は CPU (`verify_cc`) で独立検証してから `logs/scan_k<k>.found.log` に記録。
- `ext`（既定）: 29 追加ホイール、n < 9.06e23。
  `stream`: 巨大ホイール、width ≫ 6e17 で ~1.3–2×、n < 1.3e27。

### テスト

| パス | 内容 |

| `algorithms/` | 別セッションの CPU バケット篩実験（GPU 本流とは独立） |

## 限界と今後のレバー

- GTX 1660 Ti (sm_75) 単体で CC15 確定まで数時間〜十数時間。CC16 [2, 8.1e20] 全探索は ~数日。
- 篩は Barrett + 占有率 + hot/cold 分割でこの GPU の実用限界に到達。残る本質的レバーは
  **ハードウェア**（より高い整数演算スループット / メモリ帯域 / 複数 GPU・クラウド）。
- CC17+ は単一 GPU では非現実的（ブルートフォースの密度的限界）。
