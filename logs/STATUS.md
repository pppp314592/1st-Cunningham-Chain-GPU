## ボトルネック分析
- 高CCでは篩が支配的（MRはほぼ発生しない）。
- GPUが伸びない主因: **64bit `n % p` がGTX 1660 Tiで低速**（コンシューマNVIDIAは64bit整数除算が遅い）。
- CPUはハードウェア64bit除算を持つため篩で有利。
- ホイールをこれ以上大きくする（wlに29,31,43…追加）と候補は激減するが、
  wheel_n配列が指数的に肥大（>GB）して materialize 不可。→ 全列挙型ホイールの限界。

## 次に試す最適化案
1. **GPUカーネルの mod 高速化**: 64bit `n%p` を Barrett/32bit 還元に置換。
   n%p = ((cyc%p)*(wheel%p) + r%p) %p の分解＋ wheel%p 事前計算。
   r%p が64bitで残るのが課題 → Barrett 還元で軽減。
2. 追加素数上限(max_prime)のGPU向けチューニング。
3. CPU側: 旧ベースラインが少スレッドなら12スレッド化だけで数倍。

## ★A4 ストリーミングCRT (GPUオドメータ) 実装・検証 (2026-07-12 PDCA)
- **目的**: materialize 版の壁（wheel生存残基R個を配列常駐→2GBでR=256Mが上限、31追加不可）を
  撤廃し、より大きい wheel(31,47追加)で密度を下げる。
- **方式** (`src/cc_gpu_wheel.jl`): 各スレッドが自分の生存残基を**その場で合成**する。
  1. 作業番号 g → cyc と 生存index sidx に分解、2. sidx を mixed-radix(基数=各素数の生存残基数)で
  桁分解、3. 各桁が指す生存残基の**CRT寄与を加算** → 残基 r(mod wheel)。メモリO(Σ生存残基数)=数百要素。
- **3-limb 32bit篩**に拡張: r=r2·2^40+r1·2^20+r0 で **wheel<2^60** まで対応(従来2^51)。
  これにより wl=[..29,31,..,47] wheel=**6.15e17**、R=**2.07e11**(CC16)/3.34e11(CC15)を使用可能。
  ※materialize なら R×8B = 1.6TB＝不可能。ストリーミングだから成立。
- **API**: `build_wheel_streaming`, `gpu_wheel_stream_setup`, `_cc_wheel_kernel_stream128!`,
  `gpu_wheel_scan_stream128!`, `search_cc_gpu_wheel_stream128`, `_cc_wl_stream(k)`(k≥14で31,47追加)。
- **正当性(三重検証)**:
  1. `tests/test_gpu_stream.jl` 8/8 PASS: 既知CC7/8/12検出 + stream==materialize一致(CC8/12/13/15,
     オドメータ+CRT+3limbを実証) + 拡張wl(29,31追加)が同一結果。
  2. `exp_stream_verify.jl`: 既知CC15先頭が全wheel素数(31,47含む)を通過することをCPUで即時証明。
  3. `exp_stream_rediscover.jl`: **既知最小CC16 `810433818265726529159` を実機再発見 (346s, 1cyc全走査)**。
- **速度(honest)**: 大レンジ(≫wheel=6.15e17)で **stream/ext(29) ≈ 1.3〜2.0x**
  (`exp_stream_verify.jl`: density 6.08e-7→3.36e-7 = 0.55倍、スレッド速度は同程度〜1.36倍遅、
  再発見runの実測warm時 5.98e8 threads/s は ext(5.33e8)と同等)。
  ⚠**注意**: wheel が 6.15e17 と巨大なため、**レンジ幅がwheelを大きく超える実走査でのみ有効**
  (レンジ<wheel だと1周期を無駄に全列挙し逆に遅い。5e16レンジのテストでは12倍の無駄が出て10倍遅かった)。
- **結論**: 当初期待の「10倍」は非現実的だった(篩+MRが支配的でwheel追加の利得は密度比まで)。
  現実的な上積みは **大レンジ走査で ~1.3〜2x**。ただし materialize 不能な大wheelを扱える点が本質価値で、
  さらに素数を積む(53,59…, wheel<2^60内)拡張余地も確保した。ハードの壁は依然として最大要因。

## ★プロファイル → Barrett還元篩 (2026-07-12 PDCA): 篩を ~2.3倍高速化
- **プロファイル** (`exp_profile.jl`, CC16拡張wheel, range=2e16 near 9e20):
  篩=**98.4%**(23.28s) / MR=**1.6%**(0.38s)。wheel通過~12.16e9候補中、追加素数篩生存はわずか**1693**
  (生存率~1.4e-5%)。⇒**最適化対象は篩(mod演算)、MRではない**。GPUはHW整数除算が無く各`% p`が高コスト。
- **対策=Barrett還元** (`src/prime_filter.jl` `bmod(a,p,mu)`): `% p` を mul+shift に置換。
  mu=floor(2^32/p)。前提 a<2^31, 3≤p<46340 なら補正≤1回で厳密一致(安全に2回)。
- **適用範囲**: 全カーネルの篩ループ (`_cc_wheel_kernel!`/`_cc_wheel_kernel128!`/`_cc_wheel_kernel_stream128!`)。
  2-limb(materialize)/3-limb(stream)とも各項をbmodしてから合成し UInt32 桁溢れを回避。
  state に `d_mu` 追加、setup で `_barrett_mu` 計算。
- **効果**: `exp_barrett.jl` (CC16, range=2e16): 除算版15.02s → Barrett版**6.41s = 2.343x**、survivors一致(1693)。
- **正当性**: `tests/test_gpu_wheel.jl` 14/14 PASS, `tests/test_gpu_stream.jl` 8/8 PASS,
  `verify128.jl` で既知CC15 `90616211958465842219` を再発見。全て除算版と同一結果。
- **注意**: 小ジョブ(band±5e14等)はカーネルJIT+wheel構築が支配的で壁時計は不変。
  効果は篩支配の大レンジ持続走査で顕在化(実走査の regime)。

## cyc事前計算篩 (2026-07-12): 効果1.07xで不採用
- 案: 周期(cyc)ごとに `(cyc*wheel)%p` を別カーネルで事前計算しR残基で共有、候補ループから
  cyc依存項(cp, cp*wheel_mod = 2 bmod/素数)を除去。`exp_cycbase.jl` で測定。
- 結果: CC16 range=2e16 で **1.07x のみ**(6.2s→5.79s), survivors一致(1693)。
- 理由: 生存率1.4e-5%で候補は小素数で即break→ループ短く、事前計算の追加カーネル+global読み込みが
  節約分をほぼ相殺。cyc_per×nprimes配列のメモリ増も見合わない。**不採用**。
- 教訓: Barrett(2.34x)で篩の律速はbmodスループットに移り、内側ループの算術削減は早期打ち切りに
  阻まれ利得薄。篩の更なる高速化は頭打ち。残る本質的レバーはハードウェア…と思われたが、
  下記の占有率チューニング(threads=64)とhot/cold分割でさらに上積みできた。

## ★hot/cold パス分割 (篩/MR分離) (2026-07-12): さらに 1.37〜1.42x
- **仮説**: 融合カーネルは MR(128bit)コードが全スレッドのレジスタを占有(**88reg**)し、
  98.4%を占める篩の占有率を下げている。篩とMRを別カーネルに分離すれば篩の占有率↑。
- **測定** (`exp_split.jl`, CC16 range=2e16): 篩専用カーネル=**24reg** (88→24)。
  篩専用を連続実行し、生存候補(~1.4e-5%)を `d_surv` に蓄積、MRは末尾+一定タイルごとに
  ドレイン → **6.52s vs 融合8.86s = 1.37〜1.42x** (threads 64〜128)。survivors一致(1618)。
- **重要**: タイル毎にMRを挟むと非同期パイプラインにバブルが出て利得消滅(実測1.0x)。
  ホスト同期版も1.0x。**蓄積→稀にドレイン**が鍵。MRカーネルは `d_scnt` をデバイスから読み
  ホスト同期不要。ドレインは `drain_every`(既定256)タイルごと + 末尾。scap=1<<20(生存バッファ)。
- **統合**: `_cc_sieve_kernel128!`/`_cc_sieve_kernel_stream128!`(篩専用) + `_cc_mr_kernel128!`(MR専用)
  を追加。`gpu_wheel_scan128!`(ext) と `gpu_wheel_scan_stream128!`(stream) を2パス化。
  融合カーネルも残置(exp参照用)。
- **正当性**: tests 14/14 + 8/8 PASS。
  - 本番stream経路で既知CC15 `90616211958465842219` を再発見(hit=true)。
  - 本番ext経路 `gpu_wheel_scan128!` で既知CC16 `810433818265726529159` を280タイル帯域で再発見
    (hit=true, 38.5s)。**drain_every=256 を跨ぐ中間ドレイン+末尾ドレイン**を通しても取りこぼしなし。
- **累積効果** (旧既定 fused@256 比): Barrett 2.34x × 占有率 1.26x × 分割 1.4x ≈ 全体で大幅短縮。

## 篩マイクロ最適化の頭打ち分析 (2026-07-12): これ以上は不採用/HW律速
- **共有メモリ・ステージング不採用(分析)**: 素数テーブル `d_primes/d_mu/d_wheel_mod/d_pow20` の
  `[pi]` 読みはワープ内全スレッド同一アドレス=**ブロードキャスト読み**で既にL1最適。分岐する重い読みは
  `d_bad[off+np+1]`(threadごとnp異なる)だが45MBで**smemに入らない**(bit-packed版も却下済)。
  smem化すると占有率が半減(24regカーネルは1024threads/SM可能→8KB/blockで512threads/SM)。∴逆効果。
- **wheel密度は上限**: streamは wheel=6.15e17、次素数53を積むと ×53=3.26e19 > 2^60 で不可。
  extは wheel=4.22e14、素数追加で R(materialize)が2GB上限(R=256M)に当たる。両経路とも密度は最大化済。
- **結論**: 篩は Barrett+占有率+hot/cold分割 でこの GPU(GTX 1660 Ti/sm_75)の実用限界に到達。
  残る本質的レバーは**ハードウェア**(より高い整数演算スループット/メモリ帯域/複数GPU)。
